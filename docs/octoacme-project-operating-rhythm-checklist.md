# OctoAcme — Project Operating Rhythm Checklist

## Purpose
Use this checklist to define the operating rhythm for a project. It complements the day-to-day guidance in [OctoAcme — Execution & Tracking](./octoacme-execution-and-tracking.md).

## When to use
- During planning for any new initiative
- When a project changes pace, scope, or stakeholders
- When retrospectives identify coordination or reporting gaps

## Meeting plan

| Meeting | Cadence | Owner | Primary participants | Standard outputs |
|---|---|---|---|---|
| Daily standup |  | PM or delivery lead | PM, Developers, QA, TL | Blockers, dependency updates |
| Delivery sync |  | PM | PM, PdM, EM, TL, QA | Status update, risk review, decisions |
| Design / technical review |  | TL | TL, Developers, UX, Security, SRE | Design decisions, follow-up actions |
| Stakeholder update |  | PM | PM, PdM, sponsor, key stakeholders | Milestone status, risks, asks |
| Demo / review |  | PdM or PM | Delivery team, stakeholders | Feedback, acceptance notes |
| Release readiness review |  | PM | PM, QA, Security, SRE, Support, release owner | Go / no-go decision, sign-offs |
| Retrospective |  | PM | Delivery team, optional partners | Action items with owners and due dates |

## Artifact checklist
- [ ] Project one-pager is current
- [ ] Backlog / board owner is named
- [ ] Definition of Done is documented
- [ ] Risk register owner and review cadence are set
- [ ] Status reporting format is agreed
- [ ] Decision log location is agreed
- [ ] Release notes owner is named
- [ ] Support / incident communication channel is defined

## Reporting checklist
- [ ] Progress is reviewed at least weekly
- [ ] Risks and blockers are updated before the delivery sync
- [ ] Cross-team dependencies have named owners
- [ ] Success metrics are reviewed at the agreed cadence
- [ ] Action items from demos, incidents, or retros are tracked to closure

## Escalation triggers
- [ ] A committed milestone is at risk
- [ ] A dependency blocks the team for more than one business day
- [ ] Scope or acceptance criteria changed materially
- [ ] A release introduces unresolved QA, security, or operational risk
- [ ] A customer-facing issue requires coordinated communication
